# Usage Guide

Learn how to use DeepScanBot effectively for production web crawling.

---

## Basic Usage

After installing DeepScanBot via npm, use the `deepscanbot` command:

```bash
deepscanbot scan https://example.com
```

This will:
- Crawl the start page
- Follow links up to 2 levels deep
- Output results to `crawler_results.txt`

## Version Information

### View Version

Display the current version of DeepScanBot:

```bash
deepscanbot version
```

Output:
```
DeepScanBot CLI v1.0.0 (abc1234) built on 2024-01-15T10:30:00Z
```

### Version in JSON Format

Get version information in JSON format for scripts and automation:

```bash
deepscanbot version --json
```

Output:
```json
{
  "status": "success",
  "data": {
    "version": "1.0.0",
    "name": "DeepScanBot CLI",
    "git_commit": "abc1234",
    "build_date": "2024-01-15T10:30:00Z",
    "go_version": "go1.21.0",
    "platform": "linux/amd64"
  },
  "meta": {
    "timestamp": "2024-01-15T10:30:00Z",
    "command": "version",
    "duration_ms": 0
  }
}
```

### Version Information Fields

The version command provides the following information:

| Field | Description | Example |
|-------|-------------|---------|
| `version` | Semantic version number | `1.0.0` |
| `name` | CLI name | `DeepScanBot CLI` |
| `git_commit` | Git commit hash (if available) | `abc1234` |
| `build_date` | ISO 8601 build timestamp | `2024-01-15T10:30:00Z` |
| `go_version` | Go compiler version | `go1.21.0` |
| `platform` | OS/architecture | `linux/amd64` |

### Checking Version in Scripts

You can check the version in shell scripts:

```bash
#!/bin/bash
# Get version number
VERSION=$(deepscanbot version --json | jq -r '.data.version')
echo "DeepScanBot version: $VERSION"

# Check if version meets minimum requirement
MIN_VERSION="1.0.0"
CURRENT=$(deepscanbot version --json | jq -r '.data.version')

if [ "$CURRENT" != "$MIN_VERSION" ]; then
    echo "Warning: Expected version $MIN_VERSION, got $CURRENT"
fi
```

## Common Use Cases

### Single Page Crawl

Crawl only the specified URL without following any links:

```bash
# Using flag format
deepscanbot scan https://example.com --depth=0

# Using key=value format (backward compatible)
deepscanbot scan https://example.com depth=0
```

### Full Site Crawl

Crawl an entire website with greater depth and higher concurrency:

```bash
# Using flag format
deepscanbot scan https://example.com --depth=5 --concurrency=20

# Using key=value format (backward compatible)
deepscanbot scan https://example.com depth=5 concurrency=20
```

### JSON Output

Export results in JSON format for programmatic processing and integration:

```bash
# Using flag format
deepscanbot scan https://example.com --json --output=results

# Using key=value format (backward compatible)
deepscanbot scan https://example.com json=true output=results
```

### Crawl with Proxy

Route all traffic through a proxy server:

```bash
# Using flag format
deepscanbot scan https://example.com --proxy=http://127.0.0.1:8080

# Using key=value format (backward compatible)
deepscanbot scan https://example.com proxy=http://127.0.0.1:8080
```

### Content-Type Filtering

Filter which file types to download. Multiple types must be quoted as a single argument.

```bash
# Using flag format
deepscanbot scan https://example.com --content-types="text/html"
deepscanbot scan https://example.com --content-types="*/*"

# Using key=value format (backward compatible)
deepscanbot scan https://example.com content-types="text/html"
deepscanbot scan https://example.com content-types="*/*"
```

### Unique URLs Only

Avoid processing duplicate URLs:

```bash
# Using flag format
deepscanbot scan https://example.com --unique

# Using key=value format (backward compatible)
deepscanbot scan https://example.com unique=true
```

### Resume a Crawl

Stop and resume a crawl without recrawling already visited URLs:

```bash
# Initial crawl
deepscanbot scan https://example.com --depth=3 --output=my-crawl

# Resume (skips already crawled URLs)
deepscanbot scan https://example.com --depth=3 --resume --output=my-crawl
```

### Polite Crawling

Add delays to avoid overwhelming servers:

```bash
# Using flag format
deepscanbot scan https://example.com --delay=500ms --host-concurrency=2

# Using key=value format (backward compatible)
deepscanbot scan https://example.com delay=500ms host-concurrency=2
```

### Crawl with Retry and Error Handling

```bash
# Using flag format
deepscanbot scan https://docs.example.com --depth=2 --retries=3 --retry-backoff=2s --delay=1s --host-concurrency=1

# Using key=value format (backward compatible)
deepscanbot scan https://docs.example.com depth=2 retries=3 retry-backoff=2s delay=1s host-concurrency=1
```

Retries failed requests up to 3 times with exponential backoff, waits 1 second between requests to the same host, and allows only 1 concurrent request per host.

### Cross-Domain Crawl with Sitemap

```bash
# Using flag format
deepscanbot scan https://example.com --depth=3 --cross-domain --sitemap --concurrency=10 --host-concurrency=2 --json

# Using key=value format (backward compatible)
deepscanbot scan https://example.com depth=3 cross-domain=true sitemap=true concurrency=10 host-concurrency=2 json=true
```

Discovers URLs from sitemap.xml, follows links to any domain, with 10 total workers and 2 per host.

- **Backward Compatible**: Existing behavior is preserved when `--json` is not specified

#### Flag Format vs Key=Value Format

DeepScanBot supports two formats for specifying options:

**Flag Format (Recommended):**
```bash
deepscanbot scan https://example.com --depth=3 --json --output=results
```

**Key=Value Format (Backward Compatible):**
```bash
deepscanbot scan https://example.com depth=3 json=true output=results
```

Both formats work identically. Flag-based options take precedence over key=value options when both are specified.

#### Automation Use Cases

The `help --json` output enables several automation scenarios:

```bash
# Discover available commands
deepscanbot --help --json | jq '.data.root_command.subcommands[].use'

# Find flags for a specific command
deepscanbot --help --json | jq '.data.root_command.subcommands[] | select(.use | startswith("scan")) | .flags[].name'

# Check if a flag exists
deepscanbot --help --json | jq '[.data.root_command.subcommands[] | select(.use | startswith("scan")) | .flags[].name] | contains(["depth"])'

# Validate CLI interface in CI
deepscanbot --help --json | jq -e '.data.root_command.subcommands | length > 0' > /dev/null

# Generate documentation automatically
deepscanbot --help --json > cli-schema.json
```

#### Key Benefits for Automation

- **No Parsing Needed**: Consumes valid JSON directly—no need to parse human-readable help text
- **Complete Discovery**: Finds every command, subcommand, flag, and option
- **Auto-Synchronizing**: Always reflects the current CLI state as new commands and flags are added
- **Consistent Ordering**: Flags and subcommands always appear in alphabetical order for deterministic parsing
- **Stable Schema**: The JSON structure is versioned alongside the CLI, ensuring backward compatibility

### Piping and Automation

The JSON output mode is designed for easy integration with scripts and tools:

```bash
# Parse with jq
deepscanbot scan https://example.com --json | jq '.data.summary'

# Extract specific fields
deepscanbot version --json | jq -r '.data.version'

# Check command success
if deepscanbot scan https://example.com --json | jq -e '.status == "success"'; then
  echo "Scan completed successfully"
```

## Logging and Verbosity

DeepScanBot provides three logging levels to control output verbosity:

### Logging Levels

#### `--quiet`
Suppress non-essential output. Only warnings and errors are displayed.

```bash
deepscanbot scan https://example.com --quiet
```

#### `--verbose`
Display additional informational messages about the crawl progress.

```bash
deepscanbot scan https://example.com --verbose
```

#### `--debug`
Display detailed debugging information for troubleshooting.

```bash
deepscanbot scan https://example.com --debug
```

### Logging Level Precedence

When multiple logging flags are specified, they are evaluated in this order (highest priority first):

1. `--debug` (most verbose)
2. `--verbose`
3. `--quiet` (least verbose)
4. Default (info level)

**Example:**
```bash
# This will use debug mode (highest priority)
deepscanbot scan https://example.com --quiet --verbose --debug
```

### Output Streams

DeepScanBot uses separate output streams for different types of information:

- **stdout**: Command results only (JSON output, text reports)
- **stderr**: Progress messages, warnings, errors, and logs

This separation ensures that:
- Scripts can capture clean command output from stdout
- Progress information doesn't interfere with automation
- Logs are always visible in interactive terminals

**Example:**
```bash
# Capture only the JSON result
result=$(deepscanbot scan https://example.com --json)

# Redirect logs to a file
deepscanbot scan https://example.com --json 2> scan.log

# Discard logs
deepscanbot scan https://example.com --json 2>/dev/null
```

### Terminal-Aware Behavior

DeepScanBot automatically detects whether output is connected to a terminal (TTY) or a non-interactive environment:

- **Terminal (TTY)**: Full interactive experience with progress indicators
- **Non-TTY (piped/redirected)**: Automatic disabling of spinners, progress bars, and animations

This ensures that:
- Interactive users see rich progress information
- Automated scripts get clean, parseable output
- Machine-readable output (`--json`) is never polluted by UI elements

**Example:**
```bash
# Interactive terminal - shows progress bars and spinners
deepscanbot scan https://example.com

# Piped to file - no spinners or progress bars
deepscanbot scan https://example.com > output.txt

# Piped to another command - clean output
deepscanbot scan https://example.com --json | jq '.data.summary'
```

#### Quiet Mode for CI/CD

```bash
# Minimal output for CI/CD pipelines
deepscanbot scan https://example.com --quiet --no-input --force
```

#### Verbose Mode for Monitoring


## Timeout Support

DeepScanBot supports configurable timeouts for long-running operations to prevent indefinite hangs.

### Timeout Flag

The `--timeout` flag specifies the maximum duration for the entire crawl operation:

```bash
# Set timeout to 30 seconds
deepscanbot scan https://example.com --timeout=30

# Set timeout to 2 minutes
deepscanbot scan https://example.com --timeout=2m

# Set timeout to 5 minutes
deepscanbot scan https://example.com --timeout=5m
```

### Timeout Format

Timeouts accept Go duration formats:
- `30s` - 30 seconds
- `2m` - 2 minutes
- `5m` - 5 minutes
- `1h` - 1 hour

### Timeout Behavior

When a timeout is reached:
- The crawl is gracefully cancelled
- A clear error message is displayed
- The exit code is 31 (Timeout)
- A hint suggests increasing the timeout

**Example:**
```bash
$ deepscanbot scan https://example.com --timeout=5s
Error: Operation timed out after 5s.
Hint: Increase the timeout using --timeout flag (e.g., --timeout=5m).
```

### Default Timeout

The default timeout is 2 seconds. For most websites, you'll want to increase this:

```bash
# Reasonable default for most crawls
deepscanbot scan https://example.com --timeout=30s

# For large sites
deepscanbot scan https://example.com --timeout=5m
```

## stdin and File Input

DeepScanBot supports reading URLs from standard input (stdin) or files for batch processing.

### Reading from stdin

Use the `--stdin` flag to read URLs from standard input (one URL per line):

```bash
# Pipe URLs from a command
echo -e "https://example.com\nhttps://example.org" | deepscanbot scan --stdin

# Read from a file using cat
cat urls.txt | deepscanbot scan --stdin

# Generate URLs and pipe
seq 1 10 | sed 's|.*|https://example.com/page&|' | deepscanbot scan --stdin
```

### Reading from a File

Use the `--input-file` flag to read URLs from a file (one URL per line):

```bash
# Read URLs from a file
deepscanbot scan --input-file=urls.txt

# Combine with other options
deepscanbot scan --input-file=urls.txt --depth=2 --json --output=results
```

### Input File Format

The input file should contain one URL per line:

```
https://example.com
https://example.org
https://example.net/page1
https://example.net/page2
```

Empty lines are ignored.

### Input Precedence

When both `--input-file` and `--stdin` are provided, file input takes precedence:

```bash
# This will use urls.txt, not stdin
cat urls2.txt | deepscanbot scan --stdin --input-file=urls.txt
```

### Positional URL vs Input

The positional URL argument and stdin/file input are mutually exclusive:

- If you provide a positional URL (e.g., `deepscanbot scan https://example.com`), it takes precedence
- If you use `--stdin` or `--input-file`, the positional URL is ignored
- An error is returned if no input is provided

**Example:**
```bash
# Positional URL (traditional usage)
deepscanbot scan https://example.com

# stdin input (batch processing)
cat urls.txt | deepscanbot scan --stdin

# File input (batch processing)
deepscanbot scan --input-file=urls.txt
```

### Batch Processing Example

```bash
# Create a list of URLs
cat > urls.txt << EOF
https://example.com
https://example.org
https://example.net
EOF

# Process all URLs (currently scans the first one)
deepscanbot scan --input-file=urls.txt --depth=2 --json --output=batch-results

# Or pipe from another command
curl -s https://sitemap.xml | grep -o 'https://[^"]*' | deepscanbot scan --stdin
```

### Automation Example

```bash
#!/bin/bash
# batch_scan.sh - Scan multiple URLs from a file

INPUT_FILE="urls.txt"
OUTPUT_DIR="results"

mkdir -p "$OUTPUT_DIR"

while IFS= read -r url; do
    echo "Scanning: $url"
    deepscanbot scan "$url" --json --output="$OUTPUT_DIR/$(echo $url | md5sum | cut -d' ' -f1)" || {
        echo "Failed to scan: $url"
    }
done < "$INPUT_FILE"
```

## Exit Codes

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | Error (invalid arguments, crawl failure) |
| `31` | Timeout (operation exceeded configured timeout) |
