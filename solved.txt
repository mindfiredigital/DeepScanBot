DeepScanBot — Solved Issues
============================

Status: all 25 review issues are resolved.
Verification: go test ./... and go test -race ./...

Parser
------

P1
Title: Relative URLs not resolved against the page URL.
Description: Relative paths were returned unchanged and could not be fetched correctly.
Solution: Resolve references against the document base URL before returning absolute HTTP(S) URLs.

P2
Title: Parser limited to anchor and script elements.
Description: Image, stylesheet, iframe, and form links were missed.
Solution: Extract references from a, script, img, link, iframe, and form elements.

P3
Title: Non-HTTP URL schemes were not filtered.
Description: mailto:, javascript:, tel:, fragments, and empty links could be queued.
Solution: Return only valid http and https URLs.

Crawler
-------

C1
Title: WaitGroup race condition caused premature crawl completion.
Description: The dispatcher could add work after Wait saw a counter of zero.
Solution: Remove the dispatcher; the parent increments WaitGroup before it starts each child.

C2
Title: Dispatcher goroutine leaked because URL channels were never closed.
Description: The listener could block forever after each crawl.
Solution: Remove the dispatcher and its channels entirely.

C3
Title: Redundant depth guard in crawl().
Description: Depth checking was duplicated and obscured scheduling behavior.
Solution: Enforce maximum depth only while scheduling child work.

C4
Title: Raw internal Crawler struct was logged to users.
Description: Logs exposed internal synchronization state and unhelpful values.
Solution: Log readable URL, depth, concurrency, duration, and counters instead.

C5
Title: No crawl progress or final statistics.
Description: Users could not see success, failure, depth, or duration information.
Solution: Track atomic fetched, failed, and depth counters; log a completion summary.

G4
Title: No request-concurrency limit.
Description: Crawls could overwhelm target hosts with simultaneous requests.
Solution: Use a configurable request semaphore; -concurrency defaults to 10.

G5
Title: No domain scoping.
Description: Crawls followed unrelated external hosts by default.
Solution: Restrict links to the seed host and port; -cross-domain explicitly opts in to external hosts.

Fetcher
-------

F1
Title: Proxy configuration conflicted with insecure TLS configuration.
Description: One transport setting could overwrite the other.
Solution: Configure proxy and TLS on the same HTTP transport.

F2
Title: HTTP 4xx and 5xx responses were treated as successful fetches.
Description: Error pages could be parsed and saved as results.
Solution: Return an error for every 4xx and 5xx status code.

F3
Title: Requests lacked a User-Agent header.
Description: Sites could not identify the crawler.
Solution: Send User-Agent: DeepScanBot/1.0 with every request.

F4
Title: Full response bodies were read before the size limit check.
Description: Oversized pages could consume too much memory before rejection.
Solution: Use io.LimitReader to cap reads to the size limit plus one byte.

Storage and results
-------------------

S1
Title: Text output appended results across runs.
Description: Current and previous crawl results were mixed in one file.
Solution: Use os.Create to truncate the selected output file before rendering.

S2
Title: Output file was opened and closed for every URL.
Description: Large crawls incurred unnecessary per-URL filesystem operations.
Solution: Collect results, then render them once with a buffered writer.

S3
Title: JSON source metadata was embedded in a string prefix.
Description: Consumers had to split values such as "[href] https://...".
Solution: Emit URLEntry objects with url and optional source fields.

S4
Title: pageContent grew unboundedly in memory.
Description: Full response bodies remained stored for the entire crawl.
Solution: Remove pageContent and discard bodies after optional HTML link parsing.

S5
Title: GetContent was unused exported dead code.
Description: It had no call site and only served the unnecessary body cache.
Solution: Remove GetContent and the cache.

G3
Title: Tool could not be used as a library.
Description: Callers had to parse hardcoded files because Start returned no data.
Solution: Return []storage.URLEntry from Start; CLI rendering is now explicit.

CLI and architecture
--------------------

M1
Title: -h flag had no usage description.
Description: Help output did not explain the help flag.
Solution: Display “Show this help message” for -h.

M2
Title: Starting URL had no scheme validation.
Description: Invalid, relative, FTP, and file URLs failed late in the fetcher.
Solution: Validate a trimmed absolute HTTP(S) URL with a host before crawling.

M3
Title: Output filename was hardcoded.
Description: Scripts could not choose a separate output path.
Solution: Add -output as a base name and append .txt or .json.

G1
Title: No automated test coverage.
Description: The project lacked unit and integration tests.
Solution: Add coverage for parser, fetcher, storage, crawler, CLI, outputs, and robots behavior.

G2
Title: robots.txt was ignored.
Description: The crawler could request paths disallowed by site operators.
Solution: Parse and cache robots policies per origin for DeepScanBot; -ignore-robots is explicit opt-out.
